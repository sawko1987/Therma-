# Правила разработки

## Язык и именование

- Документация, чек-листы и `ADR` ведутся на русском.
- Код, идентификаторы, имена классов, файлов и переменных остаются на английском.

## Архитектура

- Бизнес-логика не живет во `Widget`-дереве.
- Каждый модуль отвечает только за свою область: `catalog`, `project`, `thermocalc`, `reporting`.
- Все внешние зависимости проходят через интерфейсы и провайдеры.

## Платформенный фокус

- Приложение проектируется в первую очередь для мобильных телефонов.
- Любые UX-решения, компоновка экранов и сценарии редактирования должны проверяться в телефонном форм-факторе.
- Локальный запуск на `Linux` используется только как среда разработки мобильного приложения, а не как отдельный desktop-режим.
- При запуске на `Linux` окно приложения должно открываться в мобильном размере по умолчанию: `412x915` логических пикселей.

## Формулы и нормативы

- Каждая расчетная формула должна иметь ссылку на источник или пункт нормы.
- Изменения расчетной логики сопровождаются тестами.
- Изменения справочников выполняются через импортируемые данные и проверяемое provenance.

## Данные

- Нельзя вручную редактировать случайные значения в нескольких местах.
- Версия датасетов хранится явно.
- Любая новая таблица или справочник должны иметь описание происхождения.

## Процесс

- `docs/checklist.md` обновляется вместе с кодом.
- Архитектурные изменения требуют обновления соответствующего `ADR`.
- Задача считается завершенной только после анализа, тестов и проверки пользовательского сценария.
