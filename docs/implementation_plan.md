# План реализации

## Этап 0. База проекта

- Инициализировать Flutter-репозиторий.
- Создать документацию, `ADR` и чек-лист.
- Зафиксировать доменную структуру и первичную навигацию.

Критерий приемки: проект запускается, документация лежит в репозитории, чек-лист заполнен стартовыми задачами.

## Этап 1. Каталоги и доменная модель

- Ввести модели климата, материалов, ссылок на нормы и проектов.
- Добавить seed-данные в `assets/data`.
- Реализовать загрузку каталогов из локальных `JSON`.

Критерий приемки: приложение отображает статистику по локальным каталогам и пример проекта.

## Этап 2. Preview thermocalc

- Собрать экран `thermocalc` в мобильной компоновке.
- Отрисовать сечение конструкции через `CustomPainter`.
- Отрисовать базовые графики по расчетным сериям.

Критерий приемки: экран показывает пример расчета, слои, графики и статусные индикаторы.

## Этап 3. Нормативный движок

- Реализовать `v1` расчета теплозащиты: сопротивление теплопередаче, температурный профиль по границам слоев, базовую нормативную проверку.
- Подготовить набор эталонных кейсов и unit/widget-тесты для расчетного модуля.
- Добавить `MVP`-скрининг влагорежима: послойное паросопротивление, базовые индикаторы риска и экранное отображение.
- Уточнять результаты по внешним публичным примерам `smartcalc.ru` в согласованных допусках без копирования их данных в приложение.

Критерий приемки: расчетные значения по теплозащите укладываются в согласованные допуски, а экран показывает связанный `MVP`-результат по влагорежиму без слома основной модели.

## Этап 4. Сохранение и отчет

- После сезонного расчета влагорежима следующий инкремент фокусируется на инфраструктуре, а не на расчетном ядре.
- Подключить локальное хранилище проектов. Статус: выполнен минимальный `E2E`-инкремент на `drift` с bootstrap demo-проекта, списком сохраненных проектов и выбором активного проекта.
- Добавить экспорт отчетов.
- Версионировать датасеты и формат проекта. Статус: выполнен базовый инкремент с явным `datasetVersion` в проекте, `drift`-миграцией схемы и авто-миграцией legacy-проектов при чтении.

Критерий приемки: проект открывается повторно, результат можно экспортировать без потери структуры.

## Этап 5. Phase 2

- Схема дома и связь элементов здания с конструкциями. Статус: выполнен рабочий инкремент с моделью помещений, ограждающих элементов, окон/дверей и конструкций внутри versioned `JSON` проекта, auto-migration legacy payload, редактированием сущностей, запуском `thermocalc` из выбранного элемента и расширенной сводкой теплопотерь по непрозрачным зонам и проёмам.
- Отопительные приборы. Статус: выполнен инкремент с моделью приборов по комнатам, локальным каталогом шаблонов, ручной корректировкой мощности, хранением в versioned `JSON` проекта и балансом покрытия теплопотерь.
- `Building heat loss v1`. Статус: выполнен отдельный экран расчета по дому, комнатам и ограждающим элементам с переиспользованием общего сервиса и учетом проёмов/отопления.
- `Ground floor v1`. Статус: выполнен отдельный persisted-модуль с экраном, моделью `groundFloorCalculations`, auto-migration legacy payload, поддержанным сценарием `плита по грунту` и тестами.
- Интеграция с `Yandex Disk`.
- Дополнительные калькуляторы.

Критерий приемки: отдельный пересмотр ТЗ после стабилизации `MVP`.
