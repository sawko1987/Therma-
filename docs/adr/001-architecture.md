# ADR 001: стартовая архитектура SmartCalc Mobile

## Статус

Принято.

## Контекст

Нужно быстро поднять работоспособный Flutter-каркас под инженерный калькулятор, не потеряв контроль над доменной моделью, источниками данных и будущими расширениями.

## Решение

- Использовать `feature-first` структуру модулей.
- Для управления состоянием использовать `flutter_riverpod`.
- Каталоги на старте грузить из локальных `JSON`-assets.
- Репозитории и расчетный движок скрывать за интерфейсами.
- Для визуализации графиков и сечения использовать `CustomPainter`.
- Полноценное локальное хранилище реализовать отдельным этапом после стабилизации доменной модели; для первого инкремента хранения использовать `drift` с versioned `JSON` payload проекта.

## Последствия

- Приложение уже сейчас запускается и показывает реальные доменные сущности без жесткой связки с конечной БД.
- Переход на `drift` выполнен без слома UI, потому что интерфейсы репозиториев были выделены заранее.
- Расчетный движок можно эволюционно развивать: текущая реализация уже закрывает теплозащиту `v1` и сезонный влагорежим, а дополнительные проверки можно добавлять без слома экрана и провайдеров.
- База `Phase 2` добавлена как семантическая модель дома внутри versioned `JSON` проекта, без немедленного перехода на отдельные таблицы или геометрический редактор.

## Примечание

Проверка `pub.dev` на `27 марта 2026` показала, что `sqlite3_flutter_libs` помечен как `EOL`. В реализации хранения используется `drift_flutter`, который поднимает native executor для `drift`; при этом миграции схемы и эволюция формата проекта остаются отдельным следующим шагом этапа 4.
