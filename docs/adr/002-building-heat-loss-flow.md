# ADR 002: шаг 2 как building-centric поток расчета

## Статус

Принято.

## Контекст

Стартовый каркас проекта был ориентирован на preview одной конструкции. После уточнения продуктового сценария стало ясно, что `шаг 2` должен работать на уровне всего объекта: комнаты, поверхности, проемы, потери по помещениям и итог по дому. При этом `шаг 3` не должен повторно собирать модель здания, а обязан читать готовый результат шага 2.

## Решение

- Считать `Project` корневым агрегатом объекта, который хранит библиотеку конструкций и список комнат.
- Ввести доменные сущности `Room`, `RoomBoundary` и `Opening`.
- Разделить сервисы на `ConstructionPerformanceEngine`, `BuildingCalculationAssembler` и `BuildingHeatLossEngine`.
- Оставить `Construction Preview` как отдельный экран детализации одной конструкции.
- Сделать `wizard` шага 2 основным пользовательским сценарием.
- Фиксировать `BuildingHeatLossResult` и `heatLossCoefficientWPerK` как стабильный контракт для шага 3.

## Последствия

- Шаг 2 больше не зависит от preview-провайдеров одной конструкции.
- Шаг 3 может строить сезонное потребление без доступа к редактору комнат и поверхностей.
- Переход на локальное хранилище можно выполнить позже без пересборки вычислительного графа.
- Construction preview сохраняется как полезный инженерный экран, но перестает быть основным маршрутом продукта.
